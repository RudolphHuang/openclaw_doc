## Context

应用当前拥有一套基于向量的记忆系统（`userMemories` 表 + pgvector HNSW 索引），通过嵌入向量进行语义相似度搜索。虽然技术上可行，但存在显著缺陷：

- **对用户不透明**：向量嵌入不可读，用户无法理解或编辑已存储的记忆
- **嵌入模型依赖**：每次记忆操作都需要调用嵌入模型生成向量
- **结构复杂**：5 张关联表（`user_memories`、`user_memories_contexts`、`user_memories_preferences`、`user_memories_identities`、`user_memories_experiences`），关系错综复杂
- **检索偏差**：向量相似度并不总能准确反映对话语境下的相关性

现有上下文引擎管道（`/packages/context-engine/`）已支持 `HistorySummaryProvider` 注入对话摘要。聊天输入区操作栏已有工具切换的 UI 模式（`/src/features/ChatInput/ActionBar/Tools/`）。用户设置页面位于 `/src/app/[variants]/(main)/settings/`。

**约束条件**：新系统必须与现有向量表共存（不做破坏性变更）。LLM 工具调用基础设施已就绪。

## Goals / Non-Goals

**目标：**
- 以人类可读的 Markdown 文本存储记忆，采用简单的两表结构（索引 + 条目）
- 由 LLM 决定记住什么、哪些记忆与当前对话相关（不依赖向量相似度）
- 提供记忆工具供 LLM 通过函数调用管理记忆
- 赋予用户完全控制权：通过管理页面查看、编辑、删除记忆
- 通过聊天界面开关实现按对话控制记忆的读写
- 支持自然语言记忆指令（"记住这个"、"忘记 X"）

**非目标：**
- 替换或迁移现有向量记忆系统（共存，非替换）
- 跨用户或共享记忆（所有记忆严格按用户隔离）
- 记忆加密或超越用户所有权的访问控制
- 自动记忆衰减或过期策略（留作未来增强）
- 记忆导入/导出功能（留作未来增强）

## Decisions

### 1. 两表数据库设计（索引 + 条目）

**决策**：使用两张表 —— `text_memories`（单条记忆）和 `text_memory_index`（按用户的索引记录）。

**理由**：索引表存储轻量的摘要列表（标题 + 一行描述），始终加载到上下文中。条目表存储完整的 Markdown 内容。这借鉴了 Claude Code 的 MEMORY.md + 独立文件模式，但适配了数据库场景。

**备选方案**：
- 单表存储所有内容：索引需要每次聚合查询，且将所有内容加载到上下文开销过大
- 基于文件的存储：不适合多用户 Web 应用的服务端渲染场景

### 2. LLM 判断相关性（无向量搜索）

**决策**：记忆索引（标题 + 描述）始终注入系统提示词。LLM 通过调用记忆工具（传入特定条目 ID）来决定加载哪些记忆的完整内容。

**理由**：LLM 在判断对话相关性方面优于余弦相似度。索引足够紧凑（约 200 条 × 每条约 100 字符 = 最多约 20K token），可以放入上下文。这彻底消除了对嵌入模型的依赖。

**备选方案**：
- 混合方案（向量预筛选 + LLM 重排序）：在预期的记忆体量下，增加复杂度但收益不明显
- 关键词搜索：对自然语言查询过于脆弱

### 3. 记忆作为函数调用工具

**决策**：将记忆操作（create、update、delete、read_detail）实现为 LLM 可调用的工具，通过现有工具系统注册。

**理由**：复用现有工具基础设施（`/src/features/ChatInput/ActionBar/Tools/`），让 LLM 明确控制何时及如何管理记忆，且记忆操作对用户可见（用户可看到 AI 何时读写记忆）。

**备选方案**：
- 后台处理（对话后提取记忆）：不够透明，增加延迟，调试困难
- 中间件调用的独立记忆 API：对用户隐藏了记忆操作

### 4. 通过聊天配置实现按对话开关

**决策**：在 Agent 聊天配置中增加 `enableMemoryRead` 和 `enableMemoryWrite` 两个布尔字段，在聊天输入区操作栏以开关形式暴露。

**理由**：完全复用现有历史消息数量开关（`/src/features/ChatInput/ActionBar/History/`）的模式。用户可能希望在敏感对话或测试场景中禁用记忆。分离读写控制提供了细粒度权限。

### 5. 记忆分类

**决策**：支持以下记忆类型：`user`（用户画像/偏好）、`feedback`（行为指导）、`project`（进行中的工作上下文）、`reference`（外部资源指针）、`general`（未分类）。

**理由**：这些分类直接映射用户需求描述，也与常见的记忆分类模式一致。`general` 作为兜底确保不遗漏信息。

### 6. 索引重建策略

**决策**：每当条目被创建、更新或删除时，索引记录都从条目表重新生成。索引本质上是条目标题和描述的物化视图。

**理由**：保持索引始终一致。在典型的记忆体量下（<500 条），通过简单 SELECT 查询重建索引足够快速，无需复杂的增量同步。

## Risks / Trade-offs

**[记忆索引占用上下文窗口]** → 限制每用户最多 200 条记忆。超出时 LLM 须合并或移除低价值条目。在管理页面显示警告。

**[记忆工具调用的 token 成本]** → 每次工具调用增加 token 消耗。通过可选开关（关闭记忆）缓解，并保持工具描述简洁。索引注入是主要成本，且已设上限。

**[记忆过时]** → 没有自动衰减机制，旧记忆可能堆积。通过访问追踪（`lastAccessedAt`、`accessedCount`）缓解，未来可增加 "过期记忆审查" 功能。管理页面允许用户手动清理。

**[并发记忆写入]** → 两个同时进行的对话可能创建冲突记忆。通过数据库级约束（每用户每分类下标题唯一）和后写者胜出语义缓解。LLM 在创建前应检查索引避免重复。

**[与向量记忆系统共存]** → 两套系统可能同时激活。新系统通过独立的表和工具运行，不与现有 `user_memories` 表交互。如果文本记忆系统验证优越，未来可规划迁移路径。
